# ADR-018: Fuzzing Target Design

## Status

Accepted

## Context

covguard parses externally-generated inputs that may be malformed or adversarial:
- **Unified diff patches**: Generated by `git diff`, may contain binary files, CRLF endings, malformed hunks
- **LCOV coverage reports**: Generated by coverage tooling, may have truncated files, invalid syntax, extreme line numbers
- **Range merging**: Mathematical operations on line ranges that must handle edge cases

Parser robustness is critical because:
1. **No panics**: A CI tool should never crash on user input
2. **Security**: Malicious inputs should not cause undefined behavior
3. **Edge cases**: Real-world diffs and coverage files contain surprising variations
4. **Confidence**: Fuzzing provides assurance beyond hand-written unit tests

Traditional unit tests cover known cases but cannot explore the vast space of possible malformed inputs. Property-based testing (proptest) helps but operates on generated structured data rather than raw byte sequences.

## Decision

We implement fuzzing using `cargo-fuzz` with the following design:

### 1. Fuzz Targets

Three fuzz targets are defined in [`fuzz/Cargo.toml`](../../fuzz/Cargo.toml):

| Target | Crate Tested | Input Type | Focus |
|--------|--------------|------------|-------|
| `fuzz_diff_parser` | `covguard-adapters-diff` | `&str` | Unified diff parsing |
| `fuzz_lcov_parser` | `covguard-adapters-coverage` | `&str` | LCOV format parsing |
| `fuzz_ranges_merge` | `covguard-ranges` | Structured | Range merge invariants |

### 2. Target Implementation Pattern

Each parser target follows the same pattern:

```rust
#![no_main]

use libfuzzer_sys::fuzz_target;
use crate_name::parse_function;

fuzz_target!(|data: &[u8]| {
    if let Ok(text) = std::str::from_utf8(data) {
        // Parser should never panic, regardless of input
        let _ = parse_function(text);
    }
});
```

Key principles:
- **No assertions**: We test for panics, not correctness
- **UTF-8 conversion**: Invalid UTF-8 is skipped; valid UTF-8 exercises the parser
- **Result ignored**: Errors are expected; panics are the failure condition

### 3. Structured Input for Range Merge

The `fuzz_ranges_merge` target uses `arbitrary` for structured input:

```rust
#[derive(Arbitrary, Debug)]
struct Input {
    ranges: Vec<(u32, u32)>,
}

fuzz_target!(|input: Input| {
    let ranges = input.ranges
        .into_iter()
        .map(|(a, b)| {
            let (start, end) = if a <= b { (a, b) } else { (b, a) };
            start..=end
        })
        .collect();
    let _ = covguard_ranges::merge_ranges(ranges);
});
```

This generates valid range structures while still exploring edge cases like empty ranges, overlapping ranges, and extreme values.

### 4. Seed Corpus

Each target has a seed corpus in `fuzz/corpus/<target>/`:

| Target | Seed Files | Purpose |
|--------|------------|---------|
| `fuzz_diff_parser` | 9 patches | Real-world diff variations (binary, CRLF, renamed, etc.) |
| `fuzz_lcov_parser` | 9 info files | LCOV format variations (empty, high counts, sparse, etc.) |
| `fuzz_ranges_merge` | 1 seed | Initial structured input |

Seed corpus files:
- Provide starting points for the fuzzer
- Document known interesting inputs
- Ensure baseline coverage from corpus execution

### 5. CI Integration

Fuzzing runs in CI with:
- Fixed seed corpus (no evolutionary fuzzing in CI)
- Timeout-based execution (e.g., 60 seconds per target)
- Failure on any panic or crash

Full evolutionary fuzzing is run manually before releases:
```bash
cargo fuzz run fuzz_diff_parser -- -max_total_time=300
cargo fuzz run fuzz_lcov_parser -- -max_total_time=300
cargo fuzz run fuzz_ranges_merge -- -max_total_time=300
```

### 6. Workspace Isolation

The fuzz crate is excluded from the main workspace:

```toml
[workspace]
members = ["."]
```

This prevents fuzz-only dependencies from affecting the main build.

## Consequences

### Positive
- **Parser robustness**: Confidence that parsers handle arbitrary input without panicking
- **Crash discovery**: Finds edge cases that unit tests miss
- **Regression prevention**: Crashing inputs can be added to corpus as permanent tests
- **Low maintenance**: Once set up, fuzzing runs automatically
- **CI safety**: Corpus-based CI runs are fast and deterministic

### Negative
- **Build complexity**: Requires nightly Rust for `cargo-fuzz`
- **Runtime cost**: Full fuzzing sessions take significant CPU time
- **False sense of security**: Fuzzing proves absence of panics, not correctness
- **Corpus management**: New crash inputs must be added to seed corpus

### Neutral
- Fuzz targets are in separate `fuzz/` directory, not in main crate
- Uses `libfuzzer-sys` crate as the fuzzing engine
- Structured fuzzing via `arbitrary` crate for complex input types
