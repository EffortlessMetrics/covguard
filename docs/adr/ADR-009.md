# ADR-009: Exit Code Strategy

## Status

Accepted

## Context

covguard is designed to run as a CI/CD gate that answers the question: "Did this PR add or change lines that are not covered by tests?" As a CLI tool intended for automation pipelines, the exit codes it returns are critical for:

- **CI/CD Integration**: Most CI systems (GitHub Actions, GitLab CI, Jenkins, etc.) use exit codes to determine pipeline success or failure
- **Scripting**: Users may wrap covguard in shell scripts that need to distinguish between different outcome types
- **Error Handling**: Callers need to differentiate between "policy failed" vs "tool broke" vs "everything is fine"
- **User Experience**: Clear exit codes help users quickly diagnose what went wrong

The challenge is choosing exit codes that are:
1. Intuitive and memorable
2. Compatible with common CI/CD conventions
3. Distinguishable between the three primary outcomes: pass, policy fail, and tool error

## Decision

We adopt a three-tier exit code strategy:

| Exit Code | Meaning | When Used |
|-----------|---------|-----------|
| `0` | Pass | All changed lines are covered, or warnings exist but `--fail-on` is not configured |
| `1` | Tool/Runtime Error | I/O errors, parse failures, invalid arguments, missing files, internal errors |
| `2` | Policy Fail | Blocking findings exist (uncovered lines, coverage below threshold) |

### Implementation Details

The exit codes are defined in [`crates/covguard-cli/src/main.rs`](../../crates/covguard-cli/src/main.rs):

```rust
/// Exit codes following the specification:
/// - 0: Pass (or warn when not fail-configured)
/// - 1: Tool/runtime error (I/O, parse failure)
/// - 2: Policy fail (blocking findings)
const EXIT_CODE_ERROR: i32 = 1;
```

### Operational Modes

The CLI supports two modes that affect exit code behavior:

1. **Standard Mode** (default): Exit codes follow the three-tier strategy above
2. **Cockpit Mode** (`--mode=cockpit`): Always exits `0` if a receipt was written successfully; exits `1` only on crash. This mode is designed for external orchestrators that consume the receipt JSON.

### Rationale for Code 2 for Policy Fail

Using exit code `2` (rather than `1`) for policy failures is intentional:

- **Convention**: Many linting and quality tools (e.g., shellcheck, some linters) use `1` for errors and higher codes for "found issues"
- **Distinction**: Using `2` allows scripts to differentiate between "the tool encountered a problem" (`1`) and "the tool worked but found issues" (`2`)
- **CI Integration**: Most CI systems treat any non-zero exit as failure, so both `1` and `2` will fail the pipeline, but the specific code provides diagnostic context

## Consequences

### Positive
- Clear semantic distinction between tool failures and policy violations
- Compatible with standard CI/CD expectations (non-zero = fail)
- Enables sophisticated shell scripting with fine-grained error handling
- Cockpit mode enables external orchestration without exit code interference
- Aligns with the error codes documented in [`docs/codes.md`](../codes.md)

### Negative
- Exit code `2` is less common than `1` for failures, may surprise some users initially
- Requires documentation for users to understand the distinction
- Cockpit mode adds complexity to exit code logic

### Neutral
- Error codes in the report JSON (e.g., `covguard.diff.uncovered_line`) are separate from process exit codes
- The `covguard explain <code>` command provides detailed information about error codes
