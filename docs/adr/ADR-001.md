# ADR-001: Hexagonal/Clean Architecture

## Status

Accepted

## Context

covguard needs to:
- Evaluate coverage policy against changed lines (pure logic)
- Integrate with multiple diff sources (git, patch files, stdin)
- Support multiple output formats (JSON, markdown, SARIF)
- Remain testable without filesystem or git dependencies

The core policy evaluation logic should not depend on external systems, making it easy to test, reason about, and adapt to new requirements.

## Decision

We adopt hexagonal architecture (ports and adapters) with three layers:

### Domain Core (`covguard-domain`)
Pure functions for policy evaluation with no side effects:
- Policy evaluation logic
- Findings generation
- Verdict computation
- Deterministic ordering

### Ports (`covguard-ports`)
Traits defining interfaces for external dependencies:
- `DiffProvider` - Source of changed lines
- `CoverageProvider` - Source of coverage data
- `RepoReader` - Repository file access
- `Clock` - Time-based operations
- `ArtifactWriter` - Output persistence

### Adapters
Implementations for specific technologies:
- `covguard-adapters-diff` - GitDiffProvider, PatchDiffProvider
- `covguard-adapters-coverage` - LcovCoverageProvider
- `covguard-adapters-repo` - FsRepoReader
- `covguard-adapters-artifacts` - FsArtifactWriter

```
┌─────────────────────────────────────────────────────────────┐
│                        CLI / App                             │
├─────────────────────────────────────────────────────────────┤
│                     Orchestrator                             │
├─────────────────────────────────────────────────────────────┤
│  Adapters (Diff)  │  Adapters (Coverage)  │  Adapters (IO)  │
├───────────────────┴───────────────────────┴─────────────────┤
│                         Ports                                │
├─────────────────────────────────────────────────────────────┤
│                      Domain Core                             │
└─────────────────────────────────────────────────────────────┘
```

## Consequences

### Positive
- Domain logic is trivially unit testable (no mocking needed)
- Easy to add new diff sources or coverage formats
- Clear separation of concerns
- Side effects are isolated to adapters
- Pure functions are easier to reason about and debug

### Negative
- More crates to maintain (20 total)
- Additional indirection through traits
- Requires discipline to keep domain pure
- Slight learning curve for new contributors

### Neutral
- Dependency injection required at orchestration layer
- Configuration needed to wire adapters to ports
