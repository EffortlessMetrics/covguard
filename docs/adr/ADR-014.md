# ADR-014: CLI Operation Modes

## Status

Accepted

## Context

covguard serves two distinct use cases that have conflicting requirements for exit code behavior:

1. **CI/CD Gate**: Traditional continuous integration where the tool's exit code determines pipeline success or failure. A non-zero exit code should block the pipeline when policy violations are found.

2. **Observability Pipeline**: Modern observability platforms (Cockpit ecosystem) that consume structured reports from multiple sensors. These orchestrators need all sensors to produce valid output regardless of findings, then aggregate and analyze results centrally.

The challenge is that a single exit code strategy cannot serve both use cases:

- For CI gates, exit code must reflect the policy verdict (pass/fail)
- For observability pipelines, exit code should only indicate whether the sensor successfully produced output

Additionally, the report schema requirements differ:

- CI mode uses `covguard.report.v1` schema focused on findings and verdicts
- Observability mode uses `sensor.report.v1` schema with input capability tracking for aggregation

## Decision

We implement two distinct CLI operation modes via the `--mode` flag:

### Standard Mode (default)

```bash
covguard check --mode=standard  # or omit --mode
```

- **Exit codes reflect verdict**:
  - `0` — Pass (all changed lines covered, or warnings exist but `--fail-on` is not configured)
  - `1` — Tool/runtime error (I/O errors, parse failures, invalid arguments)
  - `2` — Policy fail (blocking findings: uncovered lines, coverage below threshold)
- **Output schema**: `covguard.report.v1`
- **Use case**: CI/CD pipelines, pre-commit hooks, local development

### Cockpit Mode

```bash
covguard check --mode=cockpit
```

- **Exit codes indicate execution success**:
  - `0` — Receipt written successfully (regardless of findings)
  - `1` — Crash/tool error (could not produce output)
- **Output schema**: `sensor.report.v1` (extends `receipt.envelope.v1` with capabilities)
- **Use case**: Observability pipelines, external orchestrators, centralized quality dashboards

### Implementation

The mode is selected via the `CliMode` enum in [`crates/covguard-cli/src/main.rs`](../../crates/covguard-cli/src/main.rs):

```rust
#[derive(Debug, Clone, Copy, ValueEnum, Default)]
enum CliMode {
    /// Standard mode: exit based on verdict (0=pass/warn, 2=fail, 1=error)
    #[default]
    Standard,
    /// Cockpit mode: exit 0 if receipt written, exit 1 only on crash
    Cockpit,
}
```

The `sensor.report.v1` schema used in Cockpit mode includes a `capabilities.inputs` field that reports input status:

```json
{
  "schema": "sensor.report.v1",
  "run": {
    "capabilities": {
      "inputs": {
        "diff": { "status": "available" },
        "coverage": { "status": "available" }
      }
    }
  }
}
```

This enables orchestrators to distinguish between "sensor ran but found issues" vs "sensor could not access inputs."

## Consequences

### Positive

- **Clear separation of concerns**: Each mode optimizes for its target use case
- **CI-friendly defaults**: Standard mode works intuitively in traditional pipelines without configuration
- **Observability-ready**: Cockpit mode enables fleet-wide coverage monitoring without exit code interference
- **Schema consistency**: Each mode produces the appropriate schema for downstream consumers
- **Explicit selection**: The `--mode` flag makes the operating mode visible in command history

### Negative

- **Additional complexity**: Users must understand which mode to use for their scenario
- **Documentation burden**: Two modes require separate documentation and examples
- **Testing overhead**: Both modes must be tested for all scenarios

### Neutral

- Cockpit mode is opt-in; existing CI pipelines continue working unchanged
- The mode selection does not affect the core coverage evaluation logic, only exit code mapping and output schema
- ADR-009 (Exit Code Strategy) describes the three-tier exit codes for Standard mode; Cockpit mode is a separate strategy
