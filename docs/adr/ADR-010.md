# ADR-010: Error Handling Strategy

## Status

Accepted

## Context

covguard is a CLI tool that runs in CI pipelines and needs clear, actionable error handling:

- **User experience**: Developers need to quickly understand what went wrong and how to fix it
- **CI integration**: Exit codes must distinguish between policy failures (expected) and tool errors (unexpected)
- **Debugging**: Errors need enough context to diagnose issues without verbose logging
- **Stability**: Parsers (diff, LCOV) must never panic on malformed input
- **Documentation**: Error codes need stable identifiers that can be referenced in docs

The tool operates in two modes with different error handling requirements:
- **Standard mode**: Exit codes reflect verdict (0=pass/warn, 2=fail, 1=error)
- **Cockpit mode**: Always exit 0 if receipt is written, errors captured in report

## Decision

### Three-Tier Error Classification

We classify errors into three categories with distinct handling:

| Category | Prefix | Exit Code | Example |
|----------|--------|-----------|---------|
| Policy findings | `covguard.diff.*` | 2 | Uncovered lines, below threshold |
| Input errors | `covguard.input.*` | 1 | Invalid LCOV, invalid diff |
| Tool errors | `tool.*` | 1 | I/O failure, internal error |

### Error Code Format

Error codes follow a namespaced format:

```
covguard.<domain>.<specific_issue>
tool.<error_type>
```

**Policy codes** (`covguard.diff.*`):
- `covguard.diff.uncovered_line` - Changed line has zero coverage hits
- `covguard.diff.coverage_below_threshold` - Diff coverage % below threshold
- `covguard.diff.missing_coverage_for_file` - File has changes but no LCOV record

**Input codes** (`covguard.input.*`):
- `covguard.input.invalid_lcov` - LCOV parse failure
- `covguard.input.invalid_diff` - Diff parse failure

**Tool codes** (`tool.*`):
- `tool.runtime_error` - Internal/unexpected error

### Error Type Architecture

We use `thiserror` for error types with layered propagation:

```
┌─────────────────────────────────────────────────────────────┐
│                      CliError (CLI layer)                   │
│  - MissingDiffSource, ConflictingDiffSource, FileRead, etc. │
├─────────────────────────────────────────────────────────────┤
│                      AppError (App layer)                   │
│  - DiffParse, LcovParse, Io                                 │
├─────────────────────────────────────────────────────────────┤
│             Adapter Errors (DiffError, LcovError)           │
│  - Parse errors with line/column context                    │
└─────────────────────────────────────────────────────────────┘
```

Each layer:
1. Converts lower-level errors to its own type via `From` implementations
2. Adds context (e.g., file paths) at the boundary
3. Produces human-readable messages via `Display`

### Code Registry

All error codes are registered in [`CODE_REGISTRY`](../../crates/covguard-types/src/lib.rs) with:
- Stable code identifier
- Short name for programmatic use
- Descriptions (short and full)
- Remediation guidance
- Help URI linking to [`docs/codes.md`](../codes.md)

```rust
pub const CODE_REGISTRY: &[CodeInfo] = &[
    CodeInfo {
        code: CODE_UNCOVERED_LINE,
        name: "UncoveredLine",
        short_description: "Uncovered changed line",
        full_description: "A changed line has zero hits in LCOV coverage data.",
        remediation: "Add tests to execute the line, or use covguard: ignore / exclude the path.",
        help_anchor: "uncovered_line",
        help_uri: "https://github.com/EffortlessMetrics/covguard/blob/main/docs/codes.md#uncovered_line",
    },
    // ...
];
```

### Exit Code Semantics

| Code | Meaning | When Used |
|------|---------|-----------|
| 0 | Pass/Success | No blocking findings, or cockpit mode receipt written |
| 1 | Tool Error | Parse failure, I/O error, internal error |
| 2 | Policy Fail | Blocking findings (uncovered lines, below threshold) |

### Cockpit Mode Fallback

In cockpit mode, errors are captured in the report rather than causing non-zero exit:

1. Parse errors produce a report with `verdict.status: skip` and `reason: tool_error`
2. Missing inputs are recorded in `capabilities.diff` and `capabilities.coverage`
3. A fallback receipt is written even when argument parsing fails

This ensures "No Green By Omission" - the receipt always reflects what actually happened.

## Consequences

### Positive
- Clear separation between expected (policy) and unexpected (tool) failures
- Stable error codes enable documentation links and automated remediation
- `covguard explain <code>` provides inline help
- Layered error types preserve context while maintaining clean APIs
- Cockpit mode ensures CI can distinguish "tool crashed" from "policy failed"
- Parsers are fuzz-tested to never panic on malformed input

### Negative
- More error types to maintain across crate boundaries
- `From` implementations required at each layer boundary
- Error messages must be manually crafted for clarity
- Code registry must be kept in sync with `docs/codes.md`

### Neutral
- `thiserror` is the only error-handling dependency
- Exit code 2 is non-standard but necessary for CI policy gates
- Error codes are strings (not enums) for JSON serialization stability
