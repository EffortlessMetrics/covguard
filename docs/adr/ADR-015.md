# ADR-015: Multi-layered Testing Strategy

## Status

Accepted

## Context

covguard is a diff-scoped coverage gate tool that answers whether a PR added or changed lines that are not covered by tests. As a quality gate that CI pipelines depend on, the tool must be highly reliable:

- **False positives** (blocking a PR incorrectly) erode developer trust and disrupt productivity
- **False negatives** (allowing uncovered code to merge) undermine the coverage guarantee
- **Parser failures** (crashing on malformed input) break CI pipelines
- **Non-deterministic output** makes debugging difficult and breaks snapshot comparisons

The domain logic involves subtle edge cases:
- Range merging for overlapping diff hunks
- Coverage percentage calculations with boundary conditions
- Path normalization across operating systems
- Deterministic ordering for stable output

A single testing approach cannot adequately cover all these concerns. Unit tests alone cannot verify that parsers handle arbitrary malformed input, nor can they confirm that the full pipeline produces correct output.

## Decision

We adopt a six-layer testing strategy, each layer targeting specific quality attributes:

### 1. Unit Tests (inline with code)

**Purpose**: Verify individual functions and modules in isolation.

**Location**: Inline in `src/lib.rs` files using `#[cfg(test)]` modules.

**Scope**:
- Pure function behavior
- Edge cases in domain logic
- Error handling paths
- Configuration parsing

**Example**:
```rust
#[cfg(test)]
mod tests {
    use super::*;
    
    #[test]
    fn calc_coverage_pct_handles_zero_total() {
        let result = calc_coverage_pct(0, 0);
        assert_eq!(result, 0.0);
    }
}
```

### 2. Property-Based Tests (proptest)

**Purpose**: Verify invariants hold across a wide range of generated inputs.

**Location**: Embedded in crate test modules.

**Scope**:
- Range merging is idempotent: `merge(merge(ranges)) == merge(ranges)`
- Range merging produces sorted, non-overlapping output
- Coverage percentage is always 0-100%
- Finding sort order is deterministic
- Coverage merging is commutative: `merge(a, b) == merge(b, a)`

**Example**:
```rust
proptest! {
    #[test]
    fn range_merge_idempotent(ranges in prop::collection::vec(any::<Range>(), 0..100)) {
        let once = merge_ranges(&ranges);
        let twice = merge_ranges(&once);
        prop_assert_eq!(once, twice);
    }
}
```

### 3. BDD Tests (Cucumber)

**Purpose**: Verify end-to-end scenarios from business perspective.

**Location**: `bdd/features/` directory with step implementations in test code.

**Scope**:
- Full pipeline: diff + LCOV → JSON/Markdown output
- Verdict mapping: covered → pass, uncovered → fail
- Exit code behavior: 0/1/2 mapping
- Profile configurations: strict, lenient, custom
- Error handling: missing files, malformed input

**Example**:
```gherkin
Scenario: Uncovered added lines fail in strict mode
    Given a diff adding 10 lines to "src/lib.rs"
    And an LCOV report with 0% line coverage
    When covguard checks coverage with profile "strict"
    Then the verdict is "fail"
    And findings include code "covguard.diff.uncovered_line"
    And the exit code is 2
```

### 4. Fuzzing (cargo-fuzz)

**Purpose**: Ensure parsers never panic on arbitrary input.

**Location**: `fuzz/fuzz_targets/` directory.

**Scope**:
- `fuzz_diff_parser`: Unified diff parsing robustness
- `fuzz_lcov_parser`: LCOV format parsing robustness
- `fuzz_ranges_merge`: Range merging with extreme values

**Example**:
```rust
fuzz_target!(|data: &[u8]| {
    if let Ok(text) = std::str::from_utf8(data) {
        // Parser should never panic, regardless of input
        let _ = parse_patch(text);
    }
});
```

### 5. Mutation Tests (cargo-mutants)

**Purpose**: Verify test suite catches bugs in critical domain logic.

**Location**: Configured in `.cargo/mutants.toml`, focused on `covguard-domain`.

**Scope**:
- `evaluate()` - Main policy evaluation entry point
- `determine_verdict()` - Findings to verdict mapping
- `calc_coverage_pct()` - Percentage calculations
- `sort_findings()` - Deterministic ordering
- `has_ignore_directive()` - Directive parsing

**Critical**: All mutants in `covguard-domain` must be caught. A surviving mutant indicates a test gap.

### 6. Golden/Snapshot Tests (insta)

**Purpose**: Verify byte-stable output for deterministic comparisons.

**Location**: `crates/*/src/snapshots/` directories.

**Scope**:
- JSON report output (`report.json`)
- Markdown comment output (`comment.md`)
- SARIF output for GitHub Code Scanning
- Annotation formatting

**Example**:
```rust
#[test]
fn report_uncovered() {
    let report = generate_report(uncovered_scenario());
    insta::assert_json_snapshot!(report);
}
```

### Test Location Summary

| Layer | Tool | Location |
|-------|------|----------|
| Unit | built-in | `crates/*/src/lib.rs` |
| Property | proptest | `crates/*/src/lib.rs` |
| BDD | cucumber | `bdd/features/` |
| Fuzzing | cargo-fuzz | `fuzz/fuzz_targets/` |
| Mutation | cargo-mutants | `.cargo/mutants.toml` |
| Golden | insta | `crates/*/src/snapshots/` |

## Consequences

### Positive

- **High confidence in correctness**: Multiple layers catch different bug classes
- **Parser robustness**: Fuzzing ensures arbitrary input never crashes the tool
- **Refactoring safety**: Mutation tests verify test quality, not just coverage
- **Deterministic output**: Snapshot tests catch regressions in output format
- **Business alignment**: BDD tests document expected behavior in plain language
- **Invariant verification**: Property tests cover edge cases humans might miss
- **CI reliability**: A coverage gate tool must itself be highly reliable

### Negative

- **CI time impact**: Full test suite including mutation and fuzz takes significant time
- **Learning curve**: Contributors must understand multiple testing paradigms
- **Maintenance burden**: Six testing frameworks require ongoing maintenance
- **Snapshot churn**: Minor output changes require snapshot updates
- **Flaky potential**: Property and fuzz tests can occasionally find new edge cases

### Neutral

- **Selective CI**: Consider running mutation tests and extended fuzzing only on main branch or scheduled builds
- **Snapshot workflow**: `cargo insta review` becomes part of the development workflow
- **Test isolation**: Each layer runs independently, allowing parallel execution
