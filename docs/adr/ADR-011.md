# ADR-011: Configuration Precedence

## Status

Accepted

## Context

covguard needs a clear and predictable configuration system that supports multiple use cases:

- **Local development**: Developers may want project-specific defaults in a config file
- **CI pipelines**: CI jobs often need to override settings via command-line arguments
- **Quick experimentation**: Users need to test different thresholds without editing files
- **Team consistency**: Projects need shareable configuration via version control

The challenge is determining which configuration source takes priority when the same setting is specified in multiple places:

- CLI arguments (e.g., `--threshold 90`)
- Configuration file (`covguard.toml`)
- Built-in profile defaults (see ADR-007)
- Global hardcoded defaults

Without a documented precedence order, users may be confused about which settings are actually active, leading to unexpected behavior and difficult debugging.

## Decision

We implement a strict precedence hierarchy for configuration resolution:

```
CLI arguments > Config file values > Profile defaults > Global defaults
```

### Precedence Order (highest to lowest priority)

1. **CLI arguments** - Explicit command-line flags always win
   - Example: `--threshold 95` overrides all other threshold settings
   - Example: `--profile strict` overrides profile in config file

2. **Config file values** - Settings in `covguard.toml`
   - Auto-discovered from current directory upward
   - Explicitly specified via `--config path/to/covguard.toml`
   - Example: `min_diff_coverage_pct = 80` in config

3. **Profile defaults** - Built-in defaults for the selected profile
   - Profile selected via `profile = "team"` in config or `--profile team` on CLI
   - See ADR-007 for profile definitions

4. **Global defaults** - Hardcoded fallback values
   - Used only when no other source specifies a value
   - Example: `scope = "added"`, `fail_on = "error"`

### Implementation

The [`resolve_config()`](../../crates/covguard-config/src/lib.rs:259) function implements this precedence:

```rust
pub fn resolve_config(config: Option<&Config>, cli: &CliOverrides) -> EffectiveConfig {
    // Start with profile defaults or global defaults
    let profile = config.and_then(|c| c.profile).unwrap_or(Profile::Team);
    let mut effective = profile_defaults(profile);

    // Apply config file values (layer 2)
    if let Some(config) = config {
        if let Some(scope) = config.scope {
            effective.scope = scope;
        }
        // ... other config file overrides
    }

    // Apply CLI overrides (layer 1 - highest priority)
    if let Some(scope) = cli.scope {
        effective.scope = scope;
    }
    // ... other CLI overrides

    effective
}
```

### Override Behavior

Each configuration layer only overrides values it explicitly sets:

- **Null/None values do not override**: If CLI omits `--threshold`, the config file value is preserved
- **Collections are replaced, not merged**: `--path-strip` on CLI replaces the entire `path_strip` array from config
- **Profile selection is itself subject to precedence**: `--profile strict` overrides `profile = "team"` in config

### Configurable Options

| Option | CLI Flag | Config Key | Default |
|--------|----------|------------|---------|
| Profile | `--profile` | `profile` | `team` |
| Scope | `--scope` | `scope` | `added` |
| Fail threshold | `--threshold` | `min_diff_coverage_pct` | Profile-based |
| Fail mode | N/A | `fail_on` | `error` |
| Ignore directives | `--no-ignore` | `[ignore].directives` | `true` |
| Path strip | `--path-strip` | `[normalize].path_strip` | `[]` |
| Exclude patterns | N/A | `[paths].exclude` | `[]` |
| Include patterns | N/A | `[paths].include` | `[]` |

## Consequences

### Positive

- **Predictable behavior**: Users can always override config file settings via CLI without editing files
- **CI-friendly**: Pipelines can use `--threshold` to enforce stricter checks than local development
- **Debuggable**: The effective configuration is deterministic and can be traced through the precedence chain
- **No merge complexity**: Collections are replaced, avoiding confusing partial merges
- **Backward compatible**: Adding new CLI flags does not break existing config files

### Negative

- **Collection replacement**: Users cannot append to `exclude` patterns via CLI; they must repeat the full list
- **No CLI exclude/include**: Path filtering is only configurable via config file (no `--exclude` flag)
- **Debugging difficulty**: Users may not realize which config file is being used when auto-discovery is active
- **Profile interaction**: The profile selected in config file affects defaults, but CLI can override profile selection

### Neutral

- Config file auto-discovery walks up from current directory to find `covguard.toml`
- The `--config` flag can specify an explicit config file path, disabling auto-discovery
- Global defaults match the `team` profile for consistency with the default profile selection
