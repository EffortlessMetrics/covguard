# ADR-004: Unified Diff Format for Input

## Status

Accepted

## Context

covguard needs to identify which lines changed in a PR. Requirements:
- Standard format understood by developers
- Support for multiple sources (git, patch files, stdin)
- Line number information for both old and new versions
- Handle renames, additions, and deletions

Options considered:
- **Unified diff**: Universal format, git-compatible
- **Git porcelain**: Structured but git-specific
- **GitHub API**: API-dependent, rate limits
- **Custom JSON**: Maximum flexibility, no tooling

## Decision

Use unified diff format as the diff input standard.

### Supported Sources
1. **Git refs**: `--base <sha> --head <sha>` invokes `git diff --unified=0`
2. **Patch files**: `--diff-file <path>` parses unified diff directly
3. **Stdin**: Future support for piping patch directly

### Format Details
- Standard unified diff with `+++ b/<path>` headers
- Hunk markers: `@@ -old_start,old_count +new_start,new_count @@`
- Use `--unified=0` for git diffs to minimize context lines

### Parsing Model
- Extract new-side line ranges from hunks
- Handle renames (track new path identity)
- Detect and skip binary files
- Merge overlapping ranges per file

## Consequences

### Positive
- Universal format understood by all developers
- Can be generated by any VCS or diff tool
- Easy to debug (save patch, re-run locally)
- Works with GitHub PR diffs (download as patch)
- No API rate limits or authentication needed

### Negative
- Complex edge cases (renames, mode changes, binary files)
- CRLF handling complexity for Windows
- Path formats vary (a/ prefix, /dev/null for new files)
- No semantic information (moved code detection)

### Neutral
- Requires robust parser with property tests
- Fuzz testing recommended for parser security
