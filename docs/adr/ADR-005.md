# ADR-005: Schema-compliant JSON Output

## Status

Accepted

## Context

covguard produces output for consumption by:
- CI systems (exit codes, annotations)
- Cockpit systems (aggregated PR health dashboards)
- Developers (markdown comments)
- Security tools (SARIF for code scanning)

Requirements:
- Deterministic output (same inputs → same bytes)
- Schema validation for reliability
- Extensibility for tool-specific data
- Version compatibility guarantees

## Decision

Emit schema-compliant JSON following the receipt envelope pattern.

### Schema Structure
- **Envelope**: `receipt.envelope.v1.json` - Standard top-level fields for cockpit aggregation
- **Report**: `covguard.report.v1.json` - covguard-specific payload

### Required Fields
```json
{
  "schema": "covguard.report.v1",
  "tool": {
    "name": "covguard",
    "version": "0.1.0"
  },
  "run": {
    "started_at": "2026-03-12T12:00:00Z"
  },
  "verdict": {
    "status": "pass",
    "reasons": []
  },
  "findings": [],
  "summary": {}
}
```

### Determinism Rules
Findings MUST be ordered by:
1. Severity (error > warn > info)
2. Path (lexical sort)
3. Line (ascending)
4. check_id
5. Code
6. Message

### Extension Point
Tool-specific data lives under `data` at report and finding levels, allowing forward-compatible extensions.

## Consequences

### Positive
- Reliable programmatic consumption by CI tools
- Schema evolution is explicit (v1, v2, etc.)
- Easy to validate outputs in CI pipelines
- Compatible with cockpit aggregation patterns
- Deterministic output enables golden testing

### Negative
- Schema maintenance overhead
- Breaking changes require version bumps
- Additional validation step in testing
- JSON schema complexity

### Neutral
- Schema files live in `contracts/schemas/`
- Golden fixtures validate against schemas
