# ADR-016: Output Truncation Strategy

## Status

Accepted

## Context

covguard produces output in multiple formats (JSON report, markdown comments, GitHub annotations, SARIF) that are consumed by CI systems and developers. Large PRs with many uncovered lines can produce excessive output that:

- **GitHub annotations limits**: GitHub imposes limits on annotation counts per check run (typically ~50 annotations). Exceeding these limits causes silent truncation or errors.
- **Payload size limits**: Large markdown comments may exceed GitHub API payload limits or make PR comments unreadable.
- **UI readability**: Developers scanning a PR with hundreds of uncovered lines benefit from seeing the most important findings first, rather than an overwhelming list.
- **Performance**: Rendering thousands of findings impacts CI job runtime and memory usage.

The challenge is to truncate output while preserving the integrity of the verdict and providing transparency about what was omitted.

## Decision

We implement a deterministic truncation strategy with the following rules:

### 1. Sort Before Truncate

Findings are always sorted (per the determinism contract in [`contracts/rules/determinism.md`](../../contracts/rules/determinism.md)) before any truncation occurs. This ensures:
- The most severe/important findings are shown first
- Output is reproducible across runs
- Users see consistent results

### 2. Preserve Full Counts in Verdict

The `verdict.counts` object always reflects the **full** (pre-truncation) set of findings:

```json
{
  "verdict": {
    "decision": "fail",
    "reasons": ["uncovered", "truncated"],
    "counts": {
      "error": 150,
      "warning": 0,
      "total": 150
    }
  }
}
```

This ensures that policy decisions and metrics remain accurate even when the findings array is truncated.

### 3. Populate Truncation Metadata

When truncation occurs, the `data.truncation` object is populated in the report:

```json
{
  "data": {
    "truncation": {
      "findings_truncated": true,
      "shown": 50,
      "total": 150
    }
  }
}
```

The truncation metadata includes:
- `findings_truncated`: Boolean flag indicating truncation occurred
- `shown`: Number of findings actually included in the output
- `total`: Total findings before truncation

### 4. Add Truncated Reason

When findings are truncated, `"truncated"` is added to `verdict.reasons[]`. This allows consumers to programmatically detect when output was limited.

### 5. Multiple Output Budgets

Different output formats have independent limits configured via [`OutputFeatureFlags`](../../crates/covguard-output-features/src/lib.rs):

| Output Format | Default Limit | Configuration |
|---------------|---------------|---------------|
| Markdown lines | 100 | `max_markdown_lines` |
| GitHub annotations | 50 | `max_annotations` |
| SARIF results | 1000 | `max_sarif_results` |

These can be overridden via CLI flags or configuration file.

### 6. Zero Limit is Valid

`max_findings = 0` is a valid configuration that produces an empty findings array while still populating truncation metadata. This is useful for:
- Summary-only output modes
- Testing truncation behavior
- Minimal CI feedback scenarios

### 7. Omit Metadata When Not Truncated

When `max_findings` is not set or the findings count is within the limit, the `data.truncation` object is omitted entirely (not populated with `findings_truncated: false`). This keeps the output clean for the common case.

## Consequences

### Positive

- **CI stability**: Output never exceeds platform limits, preventing silent failures or truncated API responses
- **Deterministic behavior**: Same input always produces same truncated output order
- **Transparency**: Users can see exactly how many findings were omitted and why
- **Accurate metrics**: Policy decisions are based on complete data, not truncated subset
- **Flexible configuration**: Teams can tune limits based on their PR size patterns and preferences

### Negative

- **Incomplete view**: Developers must re-run locally or check the full report to see all uncovered lines
- **Configuration complexity**: Additional CLI flags and config options to understand
- **Potential confusion**: Users may not immediately understand why only N findings appear when more exist

### Neutral

- The truncation function in [`crates/covguard-output-features/src/lib.rs`](../../crates/covguard-output-features/src/lib.rs) is a pure function that returns both the truncated vector and optional metadata
- Truncation is applied independently per output format (markdown, annotations, SARIF may show different subsets)
