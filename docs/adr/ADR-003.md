# ADR-003: LCOV as Primary Coverage Format

## Status

Accepted

## Context

covguard needs to consume coverage data produced by test runners. Requirements:
- Language-agnostic format
- Wide tooling support across ecosystems
- Simple text-based format for debugging
- Line-level granularity for diff-scoped analysis

Options considered:
- **LCOV**: Text-based, widely supported, line-level data
- **Jacoco XML**: Java-specific, complex parsing
- **coverage.py JSON**: Python-specific, good structure
- **Custom JSON**: Maximum flexibility, no tooling

## Decision

Use LCOV (.info) as the primary coverage input format.

### Rationale
- **Ubiquity**: Supported by gcov, llvm-cov, coverage.py, and many other tools
- **Simplicity**: Text-based, line-oriented format easy to debug
- **Granularity**: Provides line-level hit counts (DA records)
- **Cross-language**: Works with C/C++, Rust, Python, JavaScript, and more

### Format Details
```
SF:<source_file_path>
DA:<line_number>,<hit_count>
end_of_record
```

### Implementation
- Parse SF paths and normalize to repo-relative paths
- Extract DA records for line hit counts
- Support multiple LCOV inputs with merge (max hits)
- Handle missing lines per policy configuration

## Consequences

### Positive
- Works with any language that can produce LCOV
- No need for language-specific parsers
- Well-understood format with mature tooling
- Simple to debug (text format, human-readable)
- Easy to generate test fixtures

### Negative
- LCOV format quirks (path formats vary by producer)
- No branch coverage in basic format (extensions exist)
- Some tools produce non-standard extensions
- Path normalization complexity

### Future Consideration
May add support for other formats (Jacoco, coverage.py JSON) based on user demand. The hexagonal architecture makes this straightforward—just add a new adapter.
