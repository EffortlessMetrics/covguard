# ADR-002: Multi-crate Workspace Structure

## Status

Accepted

## Context

covguard has distinct concerns that benefit from separation:
- Type definitions shared across all crates
- Domain logic with minimal dependencies
- Multiple adapters with different dependency trees
- CLI with its own dependency profile
- Need for granular versioning and publishing

A single crate would create:
- Slow compilation times
- Unclear dependency boundaries
- Difficulty in granular publishing

## Decision

Organize as a Cargo workspace with 20 focused microcrates:

### Core Crates (4)
| Crate | Responsibility |
|-------|---------------|
| `covguard-types` | DTOs, schema definitions, error codes |
| `covguard-domain` | Pure policy evaluation |
| `covguard-ports` | Trait definitions for dependencies |
| `covguard-policy` | Policy/profile model |

### Adapter Crates (4)
| Crate | Responsibility |
|-------|---------------|
| `covguard-adapters-diff` | Unified diff parsing |
| `covguard-adapters-coverage` | LCOV parsing |
| `covguard-adapters-repo` | Filesystem reading |
| `covguard-adapters-artifacts` | Artifact persistence |

### Utility Crates (4)
| Crate | Responsibility |
|-------|---------------|
| `covguard-paths` | Path normalization |
| `covguard-ranges` | Range merging logic |
| `covguard-directives` | Ignore directive parsing |
| `covguard-config` | TOML configuration |

### Output Crates (4)
| Crate | Responsibility |
|-------|---------------|
| `covguard-render` | Markdown, annotations, SARIF |
| `covguard-output` | Feature-flagged output budgets |
| `covguard-output-features` | Output feature flags |
| `covguard-reporting` | Report generation |

### Orchestration Crates (4)
| Crate | Responsibility |
|-------|---------------|
| `covguard-orchestrator` | Orchestration layer |
| `covguard-app` | Compatibility facade |
| `covguard-core` | Re-exports for convenience |
| `covguard-cli` | CLI binary |

## Consequences

### Positive
- Clear dependency boundaries enforced by compiler
- Faster incremental compilation
- Granular publishing to crates.io
- Each crate has focused, documented responsibility
- Easier to test in isolation

### Negative
- More complex workspace management
- Dependency version coordination required
- More README files to maintain
- Longer initial workspace load time

### Neutral
- Workspace inheritance used for shared metadata
- Publishing must follow topological order
